0 השורה התחתונה (קודם)

צדקת — המודל הקודם היה שטוח (שדה סכום אחד על החשבונית), ולא היה בו ישות שורות. זה לא B.P. המודל המתוקן:

כותרת אחת + שני סוגי שורות. Invoice (כותרת) ← LineItem (שורות פריט) + Payment (שורות תקבול). זהו בדיוק המבנה שרשות המסים עצמה מחייבת (רשומות C100 / D110 / D120 במבנה האחיד). ישות Invoice אחת + docType — נשארת נכונה; מה שהיה חסר אינו עוד כותרות, אלא ישויות השורות.

1 העוגן: המבנה האחיד של רשות המסים

כל תוכנת הנהלת חשבונות הרשומה בישראל חייבת לייצא את "הקובץ במבנה אחיד" (BKMVDATA) לצורכי ביקורת. זהו מודל הנתונים הרשמי למסמכים — ולכן העוגן ל-B.P שלנו. מסמך נבנה משלוש רשומות:

C100
כותרת מסמך (Header)אחת למסמך · סוג מסמך, מספר מסמך, תאריך, פרטי לקוח (שם/מזהה/כתובת), סה"כ, מע"מ, מטבע, מספר הקצאה
D110
שורת פירוט מסמך — שורת פריטרבות למסמך · מק"ט, תיאור, יחידת מידה, כמות, מחיר יחידה, % הנחה, שיעור מע"מ, סכום שורה
D120
שורת פירוט קבלה — אמצעי תשלוםרבות למסמך (בקבלה ובמס-קבלה) · סוג תשלום (מזומן/צ'ק/אשראי/העברה), סכום, בנק/סניף/חשבון, מס' צ'ק, תאריך פירעון, פרטי אשראי
שלוש מסקנות מיידיות: (1) הכותרת אחת עם שדה "סוג מסמך" לכל הסוגים → Invoice + docType מאושר. (2) שורות הפריט הן ישות נפרדת 1‎:‎רבים → LineItem. (3) קבלה אינה בנויה משורות פריט אלא משורות תקבול → Payment נפרדת. חשבונית מס-קבלה מכילה את שתיהן.

3 הטבלה המרכזית: docType → אילו שורות + הקצאה

זו נקודת הליבה — היא מסבירה למה צריך שתי ישויות שורות, והיא מניעה את לוגיקת הסוכן:

כל docType ממופה גם ל"קוד סוג מסמך" רשמי במבנה האחיד (חשבונית מס, מס-קבלה, קבלה, זיכוי…) — הקוד המספרי המדויק נעול מול הטבלה הרשמית בשלב ייצוא המבנה האחיד.

4 איפה יושבים הסכומים — ולמה השטוח נשבר

  • הסכומים חיים על השורות, ומתגלגלים לכותרת. כל LineItem: כמות × מחיר − הנחה = נטו שורה · מע"מ שורה = נטו × שיעור · סה"כ שורה. הכותרת = סכום השורות. מקור אמת יחיד = השורות (לכן שדה סכום בודד היה שגוי).
  • שיעור המע"מ הוא לפי שורה — מסמך יכול לערבב 18% עם 0% (למשל שורת יצוא/מע"מ אפס לצד שורה מקומית). מחייב vatRate ברמת השורה, לא הכותרת.
  • עיגול — הכותרת נושאת שדה עיגול (אגורות) לסכום הסופי.
  • Payment — paymentMethod (מזומן / צ'ק / אשראי / העברה / שובר) + שדות ייעודיים (מס' צ'ק+פירעון, 4 ספרות אשראי+סוג, בנק/סניף/חשבון). סכום התקבולים = הסכום ששולם.

5 לוגיקת המימוש (Remult) — המחשה בלבד, לא קוד הפרויקט

// ייכתב בפועל רק באות phase. כאן רק להראות את הלוגיקה.
@Entity('invoices')
class Invoice {
  @Fields.cuid() id
  @Field(() => DocType)        docType         // enum — כל הסוגים
  @Relations.toMany(() => LineItem)  items      // D110 — שורות פריט
  @Relations.toMany(() => Payment)  receipts   // D120 — שורות תקבול
  @Relations.toOne(() => Invoice)      linkedDoc   // עסקה→מס-קבלה, זיכוי→מקור
  @Fields.string({ allowNull: true }) allocationNumber
  @Field(() => AllocationStatus) allocationStatus
  @Field(() => InvoiceStatus)       status          // draft → open (נעילה)
  // סכומי הכותרת נגזרים מ-items+receipts — מקור אמת יחיד
}

class LineItem { description; qty; unitPrice; discountPct; vatRate; /*→*/ lineTotal }
class Payment { paymentMethod; amount; checkNo?; dueDate?; cardLast4? }

מיקום לפי קונבנציית הסיומות: invoice.entity.ts, line-item.entity.ts, payment.entity.ts, invoice.enum.ts ← shared/finance · חישוב הגלגול + קריאות TaxAuthorityManager ← server/finance.

6 שלמות חשבונאית — חוקי היסוד (קריטי)

  • אי-שינוי (Append-only): מסמך מס שהונפק הוא בלתי-ניתן-לעריכה/מחיקה בחוק. status עובר draft→open ונועל. "ביטול" = הנפקת חשבונית זיכוי מקושרת למקור. לדפר (ledger) מוסיפים, לא מוחקים.
  • רצף מספור: מספור רץ ורציף לכל סוג מסמך (חובה חוקית) — הכותרת נושאת את המספר הרץ.
  • הקצאה נעולה על הכותרת: allocationNumber נמשך פעם אחת ונשמר; לא מונפק מחדש.

7 תשואת היסודות

אם נעצב את הישויות למבנה C100/D110/D120 עכשיו, אז ייצוא המבנה האחיד (קובץ הביקורת שכל תוכנה חייבת) הופך כמעט לחינם בהמשך — נגזר ישירות מאותן ישויות דרך TaxAuthorityManager. זהו היסוד שעליו "בונים ארמון": מודל תואם-רגולציה מהיום הראשון, מודולרי, ואפס נגיעה בליבה.

פסק דין: ישות Invoice אחת + docType = B.P (כמו C100). התיקון הנדרש: הוספת LineItem (D110) ו-Payment (D120) כ-1:רבים, וסכומים שמתגלגלים מהשורות. מודל זה תואם את מה שכל מערכת חשבוניות ישראלית רשומה מיישמת.

8 שכבת הנהח"ש וייצוא לרו"ח (חשבשבת)

הבחנה קריטית: חשבשבת — כמו כל תוכנת הנה"ח של רו"ח — לא קולטת מסמכים, אלא פקודות יומן (חובה/זכות לחשבונות). במבנה האחיד אלה רשומות B100 (תנועות יומן) + B110 (חשבונות). כלומר "שידור לחשבשבת" חי שכבה אחת מתחת למסמך — ומכאן שהמסמכים לבדם לא מספיקים.

Gateway לייצוא — בדפוס TaxAuthorityManager

AccountantExportManager מאחורי Gateway, עם אדפטרים מתחלפים: (1) מבנה-אחיד (BKMVDATA מלא) · (2) ממשק הקליטה של חשבשבת — קובץ פקודות-יומן / Masofon · (3) בעתיד רמפלוס, ריווחית, ועוד. הסוכן לעולם לא כותב לפורמט ישירות — רק דרך ה-Gateway. שתי דרכי שידור לחשבשבת: (א) קובץ פקודות-יומן בפורמט הקליטה (הנפוץ), (ב) הסבה מקובץ מבנה-אחיד.

הזרימה: מסמך ← כלל רישום ← JournalEntry (+תנועות) ← AccountantExportManager ← { מבנה-אחיד · חשבשבת · … }
אותו יסוד, שני מוצאים: מהמבנה האחיד יוצאים גם קובץ הביקורת לרשות המסים וגם השידור לחשבשבת/רו"ח — בדיוק ה"מסירה לרו"ח" שסימנו כבידול. שכבה אדיטיבית נטו בצד ה-Finance, אפס נגיעה בליבה.
