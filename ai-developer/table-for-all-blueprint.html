  • מבנה server הוחזר לנעילה: server/{core · gateways · agents}. (העץ הביניים שלי השטיח את זה — תוקן.)
  • מבנה shared הוחזר לנעילה: shared/core לפי-סוג (entity/enum/controller/type) + shared/agents/<domain> בסיומות.
  • שני Gateways חדשים נכנסו למקום הנכון — server/gateways/, בדיוק בדפוס AIManager/DBManager.
  • המוצר נסגר (ב-v1 היה פתוח): "צוות AI לעסק קטן" — והישויות שלו שובצו לפי הכללים.
  • מודול ai-invoice עוגן במבנה האחיד של רשות המסים ושולב כתת-דומיין ב-finance.
  • שמות מיושרים ל-Stripe + מונחי התחום — המקור הקנוני הוא naming-conventions.html (Customer · LineItem · Payment · סטטוסי Stripe).
הנחיות עבודה (System Instructions) — מחייבות בכל סשן
  • עקרון-העל: אין גישה ישירה ל-AI / DB / מערכות חוץ מהסוכנים. הכול דרך Gateways (AIManager · DBManager · TaxAuthorityManager · AccountantExportManager).
  • אפס קוד ללא אות מפורש. אפס commit / push ללא אישור מפורש.
  • כל שאלת בחירה כוללת אפשרות מסומנת B.P.
  • שפת עבודה עברית, תשובות קצרות למובייל, עדיפות לתוצרים מוחשיים. קבצים → doc\.
  • ערך עליון: מודולריות — כל תוספת Additive, אפס נגיעה בליבה.

3 · החלטות נעולות

מ-v1, בתוספת ההכרעות של מודול ai-invoice

נושאהחלטה
ריצהמונוליט Node.js יחיד; אסינכרוניות מלאה. "מונוליט" = החלטת deployment, לא design.
עקרון-עלGateways בלבד — סוכן לא כותב שאילתות, לא פונה ל-API, לא נוגע במערכת חוץ.
AIManagerservice + adapters/ לכל ספק (grok/claude/gemini/openai), בחירה דינמית + fallback, כל קריאה ל-PromptLog.
DBManagerservice יחיד בלי adapters — ה-dataProvider של Remult הוא האדפטר.
✦ TaxAuthorityManagerGateway חדש: מספרי הקצאה מרשות המסים בזמן-אמת, ניהול הרשאת 3-החודשים, ריטריי/תור. adapters/: rashut-hamisim.
✦ AccountantExportManagerGateway חדש: ייצוא פקודות-יומן לרו"ח. adapters/: chashavshevet, bkmv-uniform (מבנה אחיד).
✦ PaymentProcessorManagerGateway חדש: סליקת אשראי מול ספקי סליקה. adapters/: tranzila, cardcom, payplus, meshulam. הסוכן קורא charge(); ההצלחה נרשמת כ-Payment.
היררכיהLayer-first (server / shared / app), ואז domain-per-agent.
servercore / gateways / agents. core = index, server, jwt, remult-api, json-body.
sharedהיברידי: core לפי-סוג (entity/enum/controller/type) + agents/<domain> בסיומות.
קריטריון הקו"האם הדפדפן מייבא?" כן → shared, לא → server.
מוסכמת שמותבתוך דומיין — סיומות (.entity/.enum/.controller/.types), לא תתי-תיקיות.
בלי registryרשימת ישויות+סוכנים ב-remult-api.ts (סגנון Remult). refactor הפיך של 10 דק' אם יידרש.
✦ מודל הנתוניםעוגן במבנה האחיד: מסמך = C100 כותרת + D110 שורות פריט + D120 שורות תקבול; הנהח"ש = B100/B110. ישות Invoice אחת + docType.
✦ שלמותמסמך מס שהונפק — append-only/נעול; תיקון רק ע"י חשבונית זיכוי מקושרת; מספור רץ.
Gitcommit / push באישור מפורש בלבד.

5 · מודל הקו — server / shared / app

האינטואיציה מאחורי כל החלטת מיקום

  • server = המטבח. לוגיקה, פרומפטים, סודות, גישה ל-DB ולמערכות חוץ. רוב-רובו של כל סוכן.
  • shared = התפריט. רק מה שמונח מול הקליינט: entity, enum מוצג, טיפוסי רשת, הצהרות controller. רזה בכוונה.
  • app = חדר האוכל. Angular קורא רק מהתפריט — לעולם לא נכנס למטבח (import מ-server אסור).
  • השם חוזר, התוכן לא. finance בשני צדי הקו = אותו נושא משתי זוויות, לא כפילות.

הבידול

  • סיווג אוטומטי של docType לפי סוג עוסק · בסיס דיווח · סכום · שולם-עכשיו · מדינת לקוח.
  • ניהול יזום של הקצאה — משיכה מראש, חידוש אוטומטי של הרשאת 3-החודשים, ריטריי, סימון "בסיכון" לפני שליחה.
  • מסירה לרו"ח — אותו יסוד מפיק קובץ ביקורת (מבנה אחיד) וגם שידור לחשבשבת (פקודות יומן).
