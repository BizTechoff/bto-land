הנחיות עבודה (System Instructions) — מחייבות בכל סשן
  • עקרון-העל: אין גישה ישירה ל-AI או ל-DB מהסוכנים. כל פעולה עוברת דרך ה-Gateways — AIManager / DBManager. לשמור על זה בכל הצעת קוד.
  • אפס קוד ללא אות מפורש מהמשתמש. אפס commit / push ללא אישור מפורש.
  • כל שאלת בחירה כוללת אפשרות מסומנת B.P (best practice).
  • שפת העבודה: עברית. תשובות קצרות וידידותיות למובייל. עדיפות לתוצרים מוחשיים (מסמכים / קבצים).
  • ערך עליון של הפרויקט: מודולריות — כל תוספת היא Additive ואינה נוגעת בליבה.

1 · תרשים הארכיטקטורה

Gateway/Service — הסוכנים הם מוחות רזים; כל משאב נגיש דרך שירות מרכזי בלבד

סוכנים — לוגיקה AIManager שכבת נתונים ספקים חיצוניים חצי הצד: קריאה/כתיבה של הסוכנים — דרך DBManager בלבד

2 · החלטות נעולות

כל מה שנסגר — עם הנימוק בקצרה

נושאהחלטה
ריצהמונוליט Node.js יחיד; אסינכרוניות מלאה (Promises) למקביליות בין סוכנים. "מונוליט" = החלטת deployment, לא design.
עקרון-עלGateways בלבד: הסוכנים לא כותבים שאילתות ולא פונים ל-APIs — צורכים שירותים קיימים.
AIManagerservice + adapters/ לכל ספק (grok, claude, gemini, openai) — כי אין ספריית איחוד ל-AI. בחירת ספק דינמית + fallback. כל קריאה מתועדת ב-PromptLog.
DBManagerservice יחיד, בלי adapters — ה-dataProvider של Remult הוא שכבת האדפטר. החלפת DB (Postgres → MySQL וכו') = שינוי שורה ב-remult-api.ts + דרייבר.
היררכיהLayer-first למעלה (server / shared / app), ואז domain-per-agent בתוך server ו-shared. נבחר על פני Feature-First ("אפשרות ב").
servercore / gateways / agents — core מכיל את הסטנדרט: index, server, jwt, remult-api, json-body.
sharedהיברידי: core לפי סוג (entity, enum, controller, type) + agents פר-דומיין ורזה.
קריטריון הקולכל קובץ: "האם הדפדפן מייבא את זה?" כן → shared, לא → server. entity/enum-מוצג/טיפוסי-רשת/הצהרת-controller → shared; לוגיקה, פרומפטים, סודות → server בלבד.
controllerמוצהר ב-shared (קריאה מהקליינט עם טיפוסים); הגוף (@BackendMethod) רץ בשרת ומאציל לשירותי server.
מוסכמת שמותבתוך דומיין סוכן — הסטנדרט כסיומות קבצים (.entity / .enum / .controller / .types), לא תתי-תיקיות.
ייבוא בין דומייניםחוזים בלבד מ-shared (interfaces), לעולם לא מימוש. מוסכמת כתיבה — ללא ESLint boundaries וללא Workspaces.
מיקרוסרוויסיםאין הכנות בשלב זה. חילוץ עתידי: תיקיות הדומיין נוסעות, החוזה נשאר ב-shared, הצרכנים מקבלים Remote Adapter על אותו interface.
סוכנים3 ל-MVP (Meeting, Marketing, Finance). סוכן נולד רק כשיש לו עבודה ודומיין; כל סוכן חדש = 2 תיקיות (server + shared), אפס נגיעה בליבה.
app ↔ סוכניםל-app תקום תיקיית סוכן רק אם יהיה לו מסך ייעודי — החלטה של שלב 5. הספירה לסוכן: 2 תמיד, 3 רק אם יש UI ייעודי.
Gitcommit / push באישור מפורש בלבד.

4 · מודל הקו — server / shared / app

האינטואיציה שמאחורי כל החלטת מיקום

  • server = המטבח. רק שם מבשלים: לוגיקה, פרומפטים, סודות, גישה ל-DB. רוב-רובו של כל סוכן.
  • shared = התפריט. רק מה שמונח על הדלפק מול הקליינט: entity, enum שמוצג, טיפוסי רשת, הצהרות controller. רזה בכוונה.
  • app = חדר האוכל. Angular. קורא אך ורק מהתפריט — לעולם לא נכנס למטבח (import מ-server אסור: קוד שרת לא מגיע לדפדפן).
  • השם חוזר, התוכן לעולם לא. תיקיית meeting בשני צדי הקו היא אותו נושא משתי זוויות — לא כפילות קוד.

6 · פריטים פתוחים

שני דברים בלבד ממתינים להכרעת המשתמש

א · מה המוצר עושה? — טרם הוחלט

השם והסוכנים הקיימים הולידו מועמדים. ההחלטה תכתיב את ה-Entities הראשונים ואת תוכן הדשבורד:
  • ניהול מסעדה — הזמנות שולחן, קמפיינים, פדיון B.P
  • ארוחות קהילתיות — שידוך זרים לשולחן משותף
  • צוות AI לעסק קטן — פגישות, שיווק, כספים לעצמאי
  • ניהול אירועים — הושבה, אישורי הגעה, תקציב
  • הזמנת עמדות Coworking
שלבים 0–2 ניטרליים למוצר — אפשר לבנות אותם לפני ההחלטה.

ב · אות להתחלת שלב 0

השלד קיים; כתיבת קוד מתחילה רק בהוראה מפורשת של המשתמש.
סטטוס נוכחי כל החלטות הארכיטקטורה נעולות. השלד קיים כ-zip אצל המשתמש (37 קבצים, כותרות בלבד). ממתינים להגדרת מוצר ולאות לשלב 0. מסמך זה מספיק לבדו כדי להמשיך את העבודה בסשן נקי.
