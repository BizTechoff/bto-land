1התמונה הגדולה

במרכז יושבת שכבת REST אחת. כל מסך במערכת הוא בעצם כתובת (route) משלו, שמותקנת על המכשיר המתאים לה — הקיוסק, הטלוויזיה בלובי, מסך הצוות, מסך המנהל. רישום נכנס דרך הקיוסק, נשמר פעם אחת, ומשם נדחף לכל המסכים בזמן אמת.

כל מטווח מקבל subdomain ייעודי תחת biztechoff — כך שאותה מערכת משרתת לקוחות רבים, כל אחד בכתובת שלו, בלי לערבב נתונים. מודולריות מלאה: מסך, שירות ועמדה הם רכיבים נפרדים שנוספים בלי כתיבה מחדש.

5לוגיקת התור

הליבה: לסרוק את הרישומים הפתוחים, למצוא את המועמד הבא לפי סוג השירות והעמדה, ולהנפיק את הכרטיס הבא.

// הבא בתור = המועמד המתאים לעמדה
let nextCandidate = null;

for (const { reg, user } of openRegistrations) {   // סריקת הרישומים
  if (reg.type === "רענון") {              // סוג השירות
    if (station.canServe(reg.type)) {         // העמדה נותנת את השירות?
      nextCandidate = reg;
      break;
    }
  }
}

const ticket = ticketAgent.getNextTicket(station);  // הנפקת/קריאת הכרטיס

// ok → store → push לכל המסכים בזמן אמת

בזרימה: המערכת מזהה את הנרשם, מציגה את מספר התור, והמדריך מקבל התראה על הבא בתור בעמדתו. אחרי אישור — הנתון נשמר ונדחף החוצה.

6מיפוי לסטאק ולמבנה שלך

איך הסקיצה יושבת על Node · Remult · Angular ועל מבנה התיקיות שלך:

shared/entity
  • Registration · הרשמה (reg)
  • Ticket · כרטיס תור
  • Station · עמדה + השירותים שהיא נותנת
  • Service · סוג שירות
  • Device · רכיב פיזי — מזהה ייחודי, סוג, שיוך
shared/enum
  • ServiceType · רענון, חידוש, אימון ראשוני, הפקדה, חנות
  • TicketStatus · ממתין / נקרא / הסתיים
shared/controller
  • QueueController · add(), getNextTicket(), callNext()
  • StationController · שיוך שירותים לעמדה
  • DeviceController · register(), pair(), disconnect()
shared/type
  • ממשקי req/res (DTOs) לכל קריאה
server/remult-api
  • Remult API + live queries · SSE (server→client) = הסנכרון בזמן אמת
  • jwt · הרשאות: מנהל / מדריך / קיוסק
app/route
  • מסלולי Angular: /kiosk /tv /lobby /state /manager /settings
  • service · repos של Remult · common · רכיבי המסכים
מודולריות: כל מסך = route + component, כל שירות/עמדה = רשומה בבסיס הנתונים. הוספה של מסך, שירות או עמדה היא תוספת — לא כתיבה מחדש.

7ניהול רכיבים · הקצאה, התחברות ואבטחה

כל מסך פיזי הוא רכיב — מכשיר אנדרואיד שמריץ את אפליקציית הקיוסק, מוגדר להצביע ללינק הראשי של הפלטפורמה, ומתחבר לפי סוגו.

מודל ההזדהות: ההתחברות היא לפי סוג — מסך מנהל מתחבר כ-manager, טלוויזיה כ-tv, קיוסק כ-kiosk, וכן הלאה. ה-role הוא הסוג. הרכיב הפיזי עצמו הוא entity (מזהה ייחודי · סוג · שיוך) שנרשם תחת ה-session של הסוג ונזכר מקומית ב-localStorage.

מחזור החיים של רכיב · Boot

1אנדרואיד → אפליקציית קיוסקמסך מלא נעול, מוגדר להצביע ללינק הראשי של הפלטפורמה.
2בדיקת localStorageיש session + רכיב נבחר? → עולה ישר לתצוגה של אותו רכיב.
3אחרת → התחברות לפי סוגמתחברים עם יוזר הסוג (manager / tv / kiosk …) — ה-role הוא הסוג.
4רשימת הרכיבים מהסוג + "הוספה"מוצגים כל הרכיבים הקיימים מהסוג הזה, עם כפתור להוספת רכיב חדש.
5מזוהה? → תצוגה · לא רשום? → רישום עצמירכיב מוכר (מה-localStorage) מופנה לתצוגה שלו; רכיב חדש רושם את עצמו ומקבל מזהה ייחודי.
6שמירה ב-localStorage / cookiesכל הבחירות נשמרות — פתיחה חוזרת מדלגת ישר לשלב 2 ולתצוגה.
7עדכונים חיים · Remult liveQuery / SSE (s2c)שאילתה בסיסית לכל role/סוג שמזינה את כל האלמנטים במסך שלו.
8כפתור התנתקות בכל רכיבמחזיר לבחירת סוג/רכיב — רק בהזנת סיסמת צוות יומית (מתחלפת) או סיסמת מנהל.

גישה לפי סוג — בלי מזהה

פנייה לכתובת כללית של סוג, למשל /tv ללא מזהה:

אם לא מחובר — יצוין שאינו מחובר ויעבור התחברות. אם מחובר — יוצגו הרכיבים הקיימים (מהסוג, או מקובצים לפי סוג בתצוגת מנהל), עם כפתור + ליד כל קבוצה להוספת רכיב מתאים.

אבטחה

ה-allowApiRead/Insert על Device מסונן ב-role — יוזר tv רואה ורושם רק רכיבי tv. שינוי שיוך והתנתקות הם פעולות מוגנות (סיסמת צוות יומית מתחלפת / סיסמת מנהל), מאומתות ב-BackendMethod.

דף הבית של העמדות (מסך הלקוח) הוא ריבועי סוגי השירות — המבקר בוחר את סוג הפנייה ומשם נכנס לתור.
