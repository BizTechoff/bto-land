01QUESTION

מהן רשימת השאלות שה-AI שואל את המשתמש?

ה-AI Intake לא שואל רשימה קבועה. הוא מבצע חילוץ מותאם הקשר: שואל רק מה שחסר כדי להריץ את ה-Eligibility Engine. רשימת השדות הקבועה היא מה שכל זכות דורשת, אבל סדר השאלות והניסוח משתנה לפי הזרימה הטבעית של השיחה.

השדות הנדרשים — לפי קבוצות

שדות זהות (חובה לכל זכות)

  1. שם מלא "מה שמך המלא?" — לרוב נחלץ אוטומטית מתעודת המילואים/אזרח ותיק שצורפה.
  2. תעודת זהות "מה מספר תעודת הזהות שלך?" — נדרש לאימות מול הרשות.
  3. טלפון נייד נחלץ אוטומטית ממספר ה-WhatsApp שממנו הגיעה ההודעה.

שדות מצב (תלויי זכות)

  1. סטטוס מילואים "אתה מחזיק כעת בתעודת משרת מילואים פעיל?" — נדרש להנחת מילואים.
  2. גיל "באיזו שנה נולדת?" — נדרש להנחת אזרח ותיק. נחלץ מת.ז אם הוקלדה.
  3. מצב משפחתי "רווק / נשוי / אלמן / גרוש?" — משפיע על שיעור ההנחה לאזרח ותיק.
  4. מצב הכנסה "הכנסתך החודשית מעל / מתחת ל-X?" — אם רלוונטי לעיר ספציפית.

שדות נכס (חובה לארנונה)

  1. רשות מקומית "באיזו עיר אתה גר?" — חובה לבחירת ה-Authority Adapter.
  2. כתובת "מה הכתובת המלאה (רחוב + מספר)?" — נדרש בטופס ההגשה.
  3. סטטוס נכס "שוכר או בעלים?" — קובע אילו זכויות חלות.
  4. מספר נכס בארנונה "יש לך מספר משלם?" — נחלץ אוטומטית מחשבון הארנונה אם הועלה.
חוקיות הזרימה ה-AI לא שואל הכל בבת אחת. הוא שואל רק שאלה אחת בכל פעם, ממתין לתשובה, ומחליט מה השאלה הבאה. אם המשתמש העלה מסמך — שדות שהמסמך כבר מספק לא נשאלים שוב.

איך זה עובד טכנית

// shared/controller/IntakeController.ts

@BackendMethod({ allowed: true })
async nextQuestion(twin: DigitalTwin): Promise<NextQ | null> {
  // אילו שדות חסרים?
  const missing = TwinSchema.requiredFields
    .filter(f => !twin[f.path]);

  if (missing.length === 0) return null;

  // המתן ל-AI שיבחר את השדה הבא לשאול
  const next = await claude.complete({
    system: INTAKE_PROMPT,
    user: { twin, missing, conversation: twin.history }
  });

  return {
    field: next.field,
    questionHe: next.naturalQuestion,
    suggestedAnswers: next.quickReplies
  };
}
02QUESTION

כיצד מפוענח הצילום שהמשתמש שולח?

כשמשתמש שולח צילום של תעודת מילואים, חשבון ארנונה, או הסכם שכירות — הוא לא הולך ל-OCR קלאסי. הוא הולך ל-Vision LLM שמבצע גם זיהוי טקסט וגם הבנה סמנטית בשלב אחד.

הזרימה המלאה

1

קבלה ושמירה

הצילום מתקבל מ-WhatsApp דרך GreenAPI webhook. נשמר ב-S3 בנתיב uploads/{userId}/{docId}.jpg, מוצפן at-rest.

2

סיווג סוג מסמך

קריאה ראשונה ל-Claude Vision: "איזה סוג מסמך זה? תעודת מילואים / חשבון ארנונה / תעודת ת.ז / אחר?". לפי התשובה — מופעל extraction schema מתאים.

3

חילוץ מובנה

קריאה שנייה ל-Claude עם schema מוגדר. למשל לתעודת מילואים: { idNumber, fullName, validFromDate, validUntilDate, signatureValid }. ה-LLM מחזיר JSON תואם, לא טקסט חופשי.

4

אימות ערכים

ולידציה: ת.ז עם בדיקת ספרת ביקורת, תאריכים בפורמט ISO, שדות חובה אינם null. אם משהו לא תקין — בקשת צילום חדש.

5

הצלבה עם Twin

אם הת.ז שחולצה לא תואמת לזו של המשתמש — דגל אזהרה. אם תוקף התעודה פג — בקשת חידוש מצה"ל.

6

אישור משתמש

סיכום מוצג למשתמש: "חולץ ת.ז 12345678, תוקף עד 31/12/2026 — לאשר?". רק אחרי אישור — ה-Twin מתעדכן.

חשוב ה-Vision LLM יכול לטעות. לכן אנחנו לא משתמשים בנתון ישירות אלא דורשים אישור מפורש מהמשתמש על כל שדה קריטי לפני שהוא נשמר ב-Twin. אם המשתמש מתקן — התיקון נשמר.

למה Vision LLM ולא OCR קלאסי?

OCR מסורתי (Tesseract, AWS Textract) מחזיר טקסט. אנחנו צריכים הבנה. הצילום של תעודת מילואים מכיל לוגו, חתימה, מספר סידורי, שם, ת.ז, תוקף. OCR יחזיר את כל הטקסט מעורבב. Vision LLM יחזיר ישירות JSON מובנה { "idNumber": "12345678", "validUntil": "2026-12-31" }, וגם יזהה אם החתימה הדיגיטלית של צה"ל נראית תקינה ויסמן זאת.

03QUESTION

איך מתבצעת ההתאמה בין ה-Twin למודול הזכות?

זה הליבה של ה-Eligibility Engine. כל מודול זכות מכיל רשימה של Predicates — תנאים אטומיים שצריכים להתקיים. ה-Engine מריץ אותם אחד-אחד מול ה-Twin ומחזיר תוצאה מובנית.

מבנה Predicate

interface Predicate {
  id: string;                 // 'has_active_miluim_certificate'
  twinPath: string;           // 'status.military.miluim.isValid'
  operator: 'eq'|'gte'|'lte'|'in'|'exists';
  expectedValue: any;
  weight: number;             // 1.0 = חובה, 0.5 = משפיע
  failureMessageHe: string;   // הודעה למשתמש אם לא מתקיים
}

אלגוריתם ההתאמה

// shared/controller/EligibilityController.ts

evaluate(twin: DigitalTwin, module: RightModule): EvalResult {
  const results = module.predicates.map(p => {
    const value = getPath(twin, p.twinPath);

    if (value === undefined) {
      return { p, status: 'MISSING', value: null };
    }

    const passes = checkOperator(value, p.operator, p.expectedValue);
    return { p, status: passes ? 'PASS' : 'FAIL', value };
  });

  // אם יש PASS על כל ה-Predicates במשקל 1.0 → ELIGIBLE
  // אם יש MISSING → INFO_NEEDED, מחזיר את השאלות החסרות
  // אם יש FAIL → NOT_ELIGIBLE, עם הסבר

  return classify(results, module);
}

דוגמה: הנחת מילואים בתל אביב

Predicate ערך ב-Twin תוצאה
has_active_miluim_certificate true ✓
municipality === 'tlv' tlv ✓
resident_status in [tenant, owner] tenant ✓
tlv_offers_discount === true true (Policy) ✓

כל ה-Predicates עברו → תוצאה: זכאי ודאי. הסכום מחושב: arnona.annualBill × 0.05 = ₪450.

חישוב מהיר ולא יקר ה-Eligibility רץ ב-פחות מ-50 מילישנייה לכל זכות. אין קריאות LLM, אין רשת — רק PostgreSQL query על Twin + הרצת Predicates ב-memory. זה מה שמאפשר לנו להציג למשתמש סיכום מהיר תוך שיחה רגילה.
04QUESTION

כיצד בפועל מתבצע העדכון על דחייה או חוסרים?

שלוש דרכים שדחייה/חוסר יכולים להגיע למערכת. שלושתן נכנסות לאותה State Machine של ה-Claim Agent.

דרך 1 — מייל מהרשות

כל הגשה נשלחת מכתובת מייל ייעודית של המערכת, למשל claim+12345@malach-hazchuyot.co.il. כשהרשות עונה, התשובה מגיעה לאותה כתובת. webhook ב-MailGun (או Postmark) מקבל את המייל, מחלץ את ה-claim ID, ומפעיל את ה-Adapter המתאים.

// server/handlers/inboundMail.ts

export async function onInboundMail(email: InboundEmail) {
  const claimId = extractClaimId(email.toAddress);
  const claim = await ClaimRepo.findById(claimId);

  // Claude מנתח: זו אישור? דחייה? בקשה למסמך?
  const intent = await classifyEmail(email.body);

  switch (intent.type) {
    case 'APPROVED':    return claim.markApproved(intent.amount);
    case 'REJECTED':    return claim.markRejected(intent.reason);
    case 'NEED_DOC':    return claim.requestDocument(intent.docType);
    case 'CLARIFY':     return claim.requestClarification(intent.question);
  }
}

דרך 2 — סקר תקופתי בפורטל

לרשויות עם פורטל מקוון (תל אביב), ה-Claim Agent מבצע polling אוטומטי כל 24 שעות. הוא מתחבר עם cookie session, ניגש לעמוד הסטטוס, ומשווה למצב הקודם.

// BullMQ job - רץ פעם ביום לכל claim פתוח
async function pollClaimStatus(claimId: string) {
  const claim = await ClaimRepo.findById(claimId);
  const adapter = getAdapter(claim.authorityId);
  const newStatus = await adapter.fetchStatus(claim.referenceId);

  if (newStatus !== claim.status) {
    await claim.recordStatusChange(newStatus);
    await notifyUser(claim);
  }
}

דרך 3 — ולידציה ב-Eligibility

זה הקישור לשאלה הקודמת. כש-EligibilityController.evaluate() מחזיר MISSING או FAIL, ה-AI Intake מציג זאת מיידית למשתמש בשיחה — עוד לפני שהבקשה הוגשה. למשל: "מצוין, אבל חסר לי תאריך תוקף התעודה. כדי לוודא שאתה זכאי, נצטרך תעודה בתוקף."

השוואה בין שלוש הדרכים

מקור זמן תגובה תיאור
מייל מהרשות מיידי (תוך דקות) הרשות עונה → webhook → Claude מנתח → עדכון Twin → הודעה למשתמש
Polling פורטל תוך 24 שעות BullMQ scheduled job בודק את הפורטל ומשווה למצב הקודם
Eligibility בזמן שיחה מיידי (50ms) במהלך ה-Intake, ה-Engine מסמן מה חסר ומה לא מתקיים
05QUESTION

מה יש בכל אחת מ-7 השכבות וכיצד מתוכנן המימוש?

כל מודול הוא npm package פנימי עם API מוגדר. החיבור ביניהם דרך dependency injection. כך אפשר להחליף מודול אחד בלי לגעת באחרים.

1. AI Intake

אחריות: שיחה עם המשתמש, חילוץ נתונים, החזרת תשובה טבעית.

מימוש: Webhook listener על GreenAPI → Session manager (Redis) → Claude API call עם system prompt → תשובה חזרה ל-WhatsApp.

API פנימי: processMessage(userId, message): Response

State: שיחה אחת = session ב-Redis עם TTL של 24 שעות. קריאות ל-Claude כוללות את ההיסטוריה.

2. Digital Twin

אחריות: ייצוג מובנה של המשתמש. JSON schema קשיח.

מימוש: Remult Entity ב-PostgreSQL. שדות מקוננים (status.military.miluim) מיוצגים כ-JSONB.

API פנימי: updateField(userId, path, value) · getTwin(userId)

גרסאות: כל שינוי נשמר ב-event log (twin_events table) — אפשר לחזור אחורה לכל מצב היסטורי.

3. Rights Catalog

אחריות: אחסון מודולי זכויות חתומים, עם versioning.

מימוש: טבלת PostgreSQL right_modules + ה-PDF החתום ב-S3. שדה signedAt + signedBy + s3Key.

API פנימי: getModule(slug, version?) · publishModule(module)

4. Eligibility Engine

אחריות: חישוב זכאות מ-Twin × Module.

מימוש: פונקציה pure ב-TypeScript, ללא state. מקבלת Twin + Module, מחזירה EvalResult. רצה ב-process.

API פנימי: evaluate(twin, module): EvalResult

5. Authority Adapters

אחריות: תקשורת עם רשות ספציפית.

מימוש: כל Adapter מממש interface אחיד AuthorityAdapter. תל אביב = Playwright browser automation. ירושלים = SMTP + תבנית מייל. חיפה = PDF generator + מייל.

API פנימי: submit(claim) | pollStatus(refId) | respondTo(refId, msg)

6. Claim Agent

אחריות: ניהול תיק לאורך זמן. State Machine + Scheduler.

מימוש: BullMQ job per claim, עם cron schedule. לכל מצב (DRAFT, SUBMITTED, FOLLOWUP) יש handler נפרד. שעון אישור־שתיקה = job עם delay של 60 יום שמופעל בעת SUBMITTED.

API פנימי: createClaim(userId, moduleId) · onClaimEvent(claimId, event)

7. Admin Panel

אחריות: ניהול מבצעי על ידי Operators.

מימוש: Angular SPA נפרד מהאתר הציבורי. Tenant: admin.malach-hazchuyot.co.il. גישה אחרי 2FA.

תפקודים: סטטיסטיקות, ניהול מודולים, חתימות עו"ד שצריך לחדש, תיקים שנעקרו ודורשים יד אנושית.

תקשורת בין מודולים כל קריאה בין מודולים עוברת דרך API פנימי מוגדר — לא דרך ייבוא ישיר של פונקציות. זה מאפשר להחליף את AI Intake מ-Claude ל-GPT, או את Authority Adapter מ-Playwright ל-API ישיר, בלי לגעת בשאר המערכת.
06QUESTION

מודול זכות — זה כמו גרסת Word חדשה?

בדיוק. כל שינוי בחוק = גרסה חדשה של המודול, בדיוק כמו שמסמך Word יכול להיות "v1, v2, v3 final, v3 final FINAL". רק שכאן זה מסודר עם semantic versioning ועם חתימה דיגיטלית של עו"ד.

איך זה עובד

כשמשרד הפנים מוציא חוזר חדש שמשנה את שיעור ההנחה למילואים מ-5% ל-7%:

v1.0.0
פרסום ראשון, 5% הנחה. תיקים פתוחים תחת גרסה זו ממשיכים בה.
DEPRECATED
v2.0.0
החוזר החדש, 7% הנחה. תיקים חדשים נפתחים תחת גרסה זו.
LIVE
v2.1.0
תוספת שיפור — Predicate חדש שתוקן אחרי משוב משפטי. בעריכה.
DRAFT

חוקי הגרסה

  1. Semantic versioning. שינוי MAJOR = שינוי משפטי משמעותי. MINOR = הוספת תנאי. PATCH = תיקון ניסוח. למשל: שינוי שיעור הנחה = MAJOR. הוספת מסמך נדרש = MINOR.
  2. גרסה אחת חיה בכל רגע. רק גרסה אחת יכולה להיות במצב LIVE בכל זמן נתון. שאר הגרסאות הן DRAFT (בעריכה) או DEPRECATED (היסטורית).
  3. תיקים פתוחים נשארים בגרסה שלהם. אם הוגש תיק תחת v1.0 ועברנו ל-v2.0, התיק ימשיך תחת v1.0 עד שייסגר. זה הגיוני משפטית: הזכות נצברה לפי החוק שהיה בתוקף בעת ההגשה.
  4. חתימה לכל גרסה בנפרד. v1.0 חתום, v2.0 דורש חתימה חדשה. גם אם השינוי הוא PATCH של תיקון ניסוח. חתימה ישנה לא תקפה לגרסה חדשה.
  5. היסטוריה מלאה. כל הגרסאות נשמרות לעד ב-S3 + DB. אם בעוד 5 שנים יבדוק עו"ד תיק היסטורי, יראה בדיוק את המודול שלפיו עבדנו אז.

Schema במערכת

// shared/entity/RightModule.ts

@Entity('right_modules')
class RightModule {
  @Fields.cuid() id: string;
  @Fields.string() slug: string;       // 'arnona-miluim'
  @Fields.string() version: string;    // '2.0.0'
  @Fields.string() status: ModuleStatus;
                                       // DRAFT|LIVE|DEPRECATED
  @Fields.json() predicates: Predicate[];
  @Fields.json() calculation: Calculation;
  @Fields.date() signedAt?: Date;
  @Fields.string() signedBy?: string;  // user ID של עו"ד
  @Fields.string() s3KeySigned?: string; // PDF חתום
  @Fields.date() effectiveFrom: Date;
  @Fields.date() effectiveUntil?: Date;
}
07QUESTION

תוקף חתימת העו"ד — מתי פג ומה אז?

חתימה דיגיטלית של עו"ד אינה פגה לפי זמן. היא תקפה כל עוד הגרסה שהיא חתומה עליה היא ה-LIVE, וכל עוד עורך הדין רשאי לעסוק במקצוע.

מתי חתימה הופכת לא תקפה?

  1. הגרסה שלה הוחלפה (DEPRECATED). כשגרסה v1.0 הוחלפה ב-v2.0, החתימה של v1.0 עוברת ל-archive. היא עדיין תקפה לתיקים היסטוריים, אך לא להגשות חדשות.
  2. החתימה הדיגיטלית נפגמה. זה נדיר אבל אפשרי — אם הקובץ ב-S3 השתנה (גם בטעות), חתימת ה-hash לא תאמת. המערכת בודקת זאת בכל קריאה ומסמנת אזהרה.
  3. עורך הדין הושעה / הוצא מהלשכה. במקרה כזה, נדרש עו"ד חליף שיחתום מחדש על כל המודולים שהיו תחת אחריותו. ה-Admin Panel מסמן זאת אוטומטית אם יש שינויי סטטוס.
  4. החוק עצמו השתנה. במקרה כזה גרסה חדשה נדרשת — וחתימה חדשה איתה.

איך החתימה נשמרת ומאומתת

1

יצירה

Operator מסיים עריכת מודול → המערכת יוצרת PDF עם כל פרטי המודול (predicates, calculation, legal sources). PDF זה משוגר לעו"ד דרך Zoho Sign.

2

חתימה

עו"ד מקבל מייל, פותח את ה-PDF, חותם דיגיטלית עם תעודה אלקטרונית מאושרת לפי חוק חתימה אלקטרונית. ה-PDF החתום חוזר אוטומטית ל-S3.

3

אחסון WORM

ה-PDF נשמר ב-S3 עם Object Lock — לא ניתן למחיקה או שינוי. רק רישום של מי גישה אליו ומתי.

4

אימות בזמן ריצה

בכל פעם שמודול נטען (להגשה או לחישוב), המערכת בודקת את ה-hash של ה-PDF, מאמתת את החתימה הדיגיטלית, ומאמתת שהעו"ד עדיין במצב פעיל.

תזכורת חשובה כש-Operator יוצר גרסה חדשה (DRAFT), הוא חייב לעבור דרך זרימת חתימה לפני שהיא הופכת ל-LIVE. אין דרך לעקוף את זה. זה אכוף ב-DB ברמת constraint: גרסה במצב LIVE חייבת להיות עם signedAt not null.

בקשה ראשונה מול השגה — הסבר מעמיק

מסלול 01

בקשת הנחה ראשונה

לפני שקיבלת חשבון ארנונה לשנת המס

אזרח שעבר לעיר חדשה, או שסטטוסו השתנה (הפך למילואימניק או לאזרח ותיק) — מגיש בקשה פרואקטיבית לרשות לפני שיוצא חשבון ארנונה ראשון. הרשות שוקלת את הבקשה ומחליטה האם להעניק את ההנחה כבר בחשבון הראשון שייצא.

תקנות הסדרים במשק המדינה (הנחה מארנונה), תשנ"ג-1993 · סעיף 2-3
כוח משפטי: בינוני. הרשות יכולה לשקול ולענות "לא" בלי שזה מפעיל מנגנון אישור־שתיקה.
מסלול 02

השגה על חשבון

אחרי שכבר קיבלת חשבון ארנונה בלי ההנחה שמגיעה

אזרח קיבל חשבון ארנונה ולא צוינה בו ההנחה שמגיעה לו. הוא מגיש השגה למנהל הארנונה, מבקש לתקן. כאן הרשות חייבת לענות תוך 60 יום (עד 90 עם הארכה). אם שתקה — ההשגה התקבלה אוטומטית לפי החוק.

תקנה 3ו לתקנות הסדרים במשק המדינה · חוק הרשויות המקומיות (ערר על קביעת ארנונה כללית), תשל"ו-1976
כוח משפטי: חזק מאוד. שעון 60 יום + חזקת קבלה אוטומטית.

// shared/controller/ClaimController.ts

selectStrategy(twin: DigitalTwin, module: RightModule): Strategy {
  // האם יש חשבון ארנונה אחרון בידינו?
  const lastBill = twin.documents.findLatest('arnona_bill');

  if (!lastBill) {
    // אין חשבון - מסלול בקשה ראשונה
    return new InitialRequestStrategy(module);
  }

  // יש חשבון - מה ההנחה הקיימת בו?
  const existingDiscount = analyzeBill(lastBill, module.slug);
  // → { exists: true, percentage: 3, amount: 270, period: '2026' }
  // או → { exists: false }

  // מה ההנחה שמגיעה לפי המודול?
  const entitledDiscount = computeEntitled(twin, module);
  // → { percentage: 5, amount: 450, period: '2026' }

  // השוואה — לא רק קיום, אלא תאימות מלאה
  const gap = entitledDiscount.amount - (existingDiscount.amount || 0);

  if (gap <= 0 && existingDiscount.exists) {
    // ההנחה תואמת או טובה יותר - אין מה להגיש
    return null;
  }

  if (existingDiscount.exists && gap > 0) {
    // יש הנחה חלקית - השגה על ההפרש
    return new PartialDiscountAppealStrategy(module, lastBill, gap);
  }

  // אין הנחה כלל - השגה רגילה
  return new AppealStrategy(module, lastBill);
}

// shared/services/BillAnalyzer.ts

async function analyzeBill(
  bill: Document, moduleSlug: string
): Promise<ExistingDiscount> {
  // Claude Vision מחלץ את כל סעיפי ההנחה מהחשבון
  const lineItems = await claudeVision.extract(bill, {
    schema: BillLineItemsSchema
  });

  // סינון לפי סוג ההנחה הרלוונטי למודול
  const relevant = lineItems.filter(item =>
    matchesDiscountType(item, moduleSlug)
  );

  if (relevant.length === 0) return { exists: false };

  return {
    exists: true,
    percentage: relevant[0].discountPct,
    amount: relevant[0].discountAmount,
    period: relevant[0].coveredPeriod,
    rawText: relevant[0].rawDescription
  };
}

function computeEntitled(
  twin: DigitalTwin, module: RightModule
): EntitledDiscount {
  // המנוע של החישוב לפי הנוסחה שב-RightModule
  const baseAmount = twin.arnona.annualBill;
  const policy = module.policyOverrides[twin.municipality];
  const rate = policy?.ratePercentage ?? module.calculation.rate;

  return {
    percentage: rate * 100,
    amount: baseAmount * rate,
    period: getCurrentTaxYear()
  };
}

10QUESTION

רישום מאגר ומינוי DPO — איך בפועל?

לפי חוק הגנת הפרטיות התשמ"א-1981 ותקנות הגנת הפרטיות (אבטחת מידע) תשע"ז-2017, מי שמנהל מאגר מידע אישי חייב ברישום ובסטנדרטים מוגדרים. הנה התהליך המלא.

שלב 1 — רישום המאגר

  1. זיהוי המאגר. "מאגר נתוני לקוחות מלאך הזכויות" — מכיל ת.ז, שם, כתובת, סטטוס מילואים/אזרח ותיק, חשבונות ארנונה, התכתבויות עם רשויות. זה מאגר שמכיל מידע רגיש לפי החוק (פרט לבריאות).
  2. הגשת בקשה לרישום. טופס מקוון באתר רשם מאגרי המידע (ברשות הגנת הפרטיות, משרד המשפטים). מילוי: סוג מאגר, מטרת השימוש, מקורות המידע, אופן השמירה, אופן ההעברה.
  3. שלם אגרה ויקבלת מספר רישום. אגרה שנתית. אחרי הרישום, חובה לציין את מספר המאגר בכל פנייה ללקוח (במדיניות הפרטיות באתר).
  4. חידוש שנתי. אם יש שינויים מהותיים (מטרות חדשות, סוגי נתונים נוספים) — חובה לעדכן.

שלב 2 — מינוי DPO (Data Protection Officer)

חובת מינוי DPO חלה על מאגרים גדולים (מעל 10,000 אנשים) או על מאגרים שמטפלים במידע רגיש במיוחד. במקרה שלנו — סביר שזה יהיה רלוונטי תוך שנה ראשונה.

  1. בחירת אדם. ה-DPO צריך להיות עצמאי — לא חבר הנהלה. יכול להיות חיצוני (יועץ פרטיות במשרת חלקית) או פנימי. בעלות נמוכה: יועץ פרטיות חיצוני ב-₪2,000-4,000 לחודש.
  2. מתן סמכויות. ל-DPO יש גישה לכל הנתונים, סמכות לדרוש תיקונים, ויכולת להגיש דוחות ישירות לרשות הפרטיות.
  3. פרסום פרטים. שם וכתובת מייל של ה-DPO מתפרסמים באתר תחת מדיניות הפרטיות, ומסופקים לכל לקוח שדורש.
  4. ביצוע ביקורות תקופתיות. כל 12 חודשים, ה-DPO מבצע audit פנימי וכותב דוח להנהלה.

שלב 3 — סטנדרטי האבטחה הנדרשים

חובה מימוש בפלטפורמה
הצפנה במנוחה PostgreSQL + S3 = AES-256 ברירת מחדל. גם backups מוצפנים.
הצפנה בתעבורה HTTPS בלבד (TLS 1.3). HSTS preload. אין כניסה ב-HTTP.
בקרת גישה JWT + refresh tokens. 2FA חובה ל-Operators. RBAC ברמת שאילתה.
audit log טבלת access_log — כל גישה ל-Twin של משתמש מתועדת: מי, מתי, מה ראה, מאיזה IP.
נוהל מחיקה endpoint POST /me/delete — מוחק את כל הנתונים תוך 30 יום, עם רישום מבצע המחיקה.
ניהול תקריות נוהל כתוב לתקריות אבטחה. דיווח לרשות הפרטיות תוך 24 שעות במקרה של דליפה.

שלב 4 — מדיניות פרטיות וכתב הסכמה

בכל שיחה ראשונה עם משתמש חדש, ה-AI Intake מציג את מדיניות הפרטיות בצורה מקוצרת: "כדי לסייע לך, אצטרך לאסוף ת.ז וכמה פרטים נוספים. הנתונים מאוחסנים מאובטחים ומשמשים רק להגשת זכויות בשמך. תוכל למחוק אותם בכל זמן. המסכים?"

ההסכמה נשמרת ב-DB עם timestamp + IP + גרסת מדיניות הפרטיות שהוצגה. כל שינוי במדיניות = הסכמה מחודשת.

חשוב להבין הלקוח (אלימלך שחר בן ארי) הוא בעל המאגר מבחינה משפטית. BizTechoff היא הספק הטכנולוגי. הרישום ברשם מאגרי המידע מתבצע ע"י הלקוח. BizTechoff מספקת את התשתית, אבל ההגשה הפורמלית והתשלום השנתי הם באחריות הלקוח. זה נרשם בתנאי ההצעה.
11QUESTION

איך מסומן הגבול לעו"ד אנושי?

ההגשה האוטומטית של בקשות והשגות מנהליות לא נחשבת לייצוג משפטי. אבל יש שלבים בהם חובה שעו"ד אנושי יטפל. כל מודול זכות מסמן את הגבולות הללו במפורש, וה-Claim Agent עוצר אוטומטית כשמגיע אליהם.

הגבולות בפועל

1

השגה ראשונה למנהל הארנונה — אוטונומי

הליך מנהלי טהור. המערכת מטפלת מקצה לקצה. אין צורך בעו"ד אנושי. ה-flag autoHandled: true.

2

ערעור לוועדת הערר — אוטונומי + התראה

גם זה הליך מנהלי, ה-Claim Agent יכול להגיש את הערעור. אבל ה-flag requiresLawyerNotification: true מפעיל מנגנון: עו"ד הליווי של המערכת מקבל התראה ויכול להעיף מבט. זה הליך כמעט אוטונומי, עם אופציית escape hatch.

3

עתירה לבית משפט — חובה עו"ד אנושי

פה המערכת עוצרת לחלוטין. ה-flag requiresHumanLawyer: true חוסם את ה-Claim Agent מלהמשיך. במקום זאת, מתבצעות הפעולות הבאות:

  • נשלחת הודעה למשתמש: "התיק שלך הגיע לשלב שבו יש לפנות לבית משפט. זה דורש ייצוג משפטי. אנו ממליצים על עו"ד הבאים:..."
  • נוצר תיעוד מלא של ההתכתבויות עד כה, מאוחסן ב-S3, וניתן להעביר לעו"ד.
  • נשלחת הודעה ל-Operator שמסמנת את התיק כ-HUMAN_LAWYER_REQUIRED.
  • שעון אישור־שתיקה נעצר עד שמתקבלת הודעה על המשך הליכים.

הזרימה האוטומטית — בעת דחייה

1

זיהוי הדחייה

תשובת הרשות מגיעה (במייל / בפורטל). Claude API מנתח את הטקסט ומסווג: { type: 'REJECTED', reason: '...', citedRegulation: '...' }. סוג הסיבה משפיע על אסטרטגיית הערעור.

2

ניתוח עילת הדחייה

ה-Claude מסווג את הסיבה לאחת מ-5 קטגוריות: חוסר מסמך (אפשר להשלים ולהגיש מחדש), פרשנות שגויה של החוק (ערעור משפטי), סיבה פרוצדורלית (ניתן לתקן), חריגת סמכות (פנייה לנציב), סיבה מהותית (יידרש בית משפט).

3

הצעה למשתמש

ה-AI שולח הודעה: "קיבלתי את התשובה מהעירייה. הם דחו בנימוק X. לדעתי הדחייה לא מוצדקת כי Y. אני יכול להגיש ערר לוועדת הערר תוך 30 יום — להמשיך?"

4

הכנת מסמך הערר

אם המשתמש מאשר, Claude מכין כתב ערר עם מבנה משפטי מקובל: עובדות, רקע משפטי, נימוקי הערר, ובקשה. כל ההתכתבויות הקודמות מצורפות כראיה.

5

חתימת עו"ד הליווי

כתב הערר עובר לעו"ד הליווי דרך Zoho Sign. זה לא ייצוג — העו"ד מאשר את התוכן ואת הניסוח. שכר טרחה: ₪150-300 לערר, מסובסד מהמסלול הבסיסי או נגבה מהמשתמש.

6

הגשה ושעונים מקבילים

הערר נשלח לוועדת הערר תוך 30 יום מקבלת הדחייה (זה הדדליין החוקי לעורר). מרגע שהערר התקבל, הרשות חייבת להגיב תוך 30 יום נוספים (תקנות סדרי דין בוועדת ערר, תקנה 4). חשוב לציין: אין מנגנון אישור־שתיקה לוועדת ערר — זה ייחודי להליך ההשגה הראשונה (תקנה 3ו). לכן השעון של ועדת הערר הוא שעון מעקב מנהלי, לא משפטי. ה-Claim Agent עוקב, שולח תזכורות, אבל לא יכול להכריז על "זכייה" אם הוועדה שותקת.

הקוד — State Machine מורחב

// shared/entity/ClaimCase.ts

enum ClaimStage {
  DRAFT,
  SUBMITTED,
  ACKNOWLEDGED,
  FOLLOWUP,
  SILENT_APPROVAL_TRIGGERED,
  APPROVED,
  REJECTED,
  // ↓ שכבות הערעור
  REJECTION_ANALYZED,        // Claude ניתח את הסיבה
  APPEAL_PROPOSED,           // הוצע למשתמש
  APPEAL_PREPARED,           // כתב הערר מוכן
  APPEAL_LAWYER_REVIEW,      // אצל עו"ד לאישור
  APPEAL_SUBMITTED,          // הוגש לוועדת ערר
  APPEAL_FOLLOWUP,           // ממתין לתגובה
  APPEAL_APPROVED,           // הערר התקבל!
  APPEAL_REJECTED,           // הערר נדחה
  HUMAN_LAWYER_REQUIRED,     // ↑ פה המערכת עוצרת
  CLOSED
}

// המעבר אוטומטי בין מצבים
function onRejection(claim: ClaimCase) {
  const reason = claim.rejectionReason;
  const strategy = analyzeRejection(reason);

  if (strategy.requiresHumanLawyer) {
    return claim.transitionTo('HUMAN_LAWYER_REQUIRED');
  }

  if (strategy.canRetryDirectly) {
    return claim.transitionTo('DRAFT', { resetWithFix: true });
  }

  // ערעור אוטומטי
  return claim.transitionTo('REJECTION_ANALYZED', {
    appealStrategy: strategy,
    deadlineDays: 30
  });
}

שעון דדליין הגשה — לא אישור־שתיקה

30ימים להגשת ערר

מרגע קבלת הדחייה — 30 יום בלבד להגיש ערר לוועדת הערר. זה דדליין שלילי — אם פספסת, איבדת את הזכות לערר. השעון רץ ב-BullMQ ושולח התראות למשתמש ב-T+7, T+14, ו-T+25 אם לא אישר את הערעור. חשוב להבחין: זה לא שעון אישור־שתיקה (שקיים רק בהשגה הראשונה לפי תקנה 3ו) אלא שעון שמחייב את האזרח לפעול בזמן.

מפת כל השעונים בתהליך

כדי שלא יהיה בלבול — להלן ריכוז של כל השעונים שהמערכת מנהלת לאורך מסע הזכות, מהרגע שקיבלת חשבון ארנונה ועד למימוש או להעברה לעו"ד אנושי.

שעון משך סוג אם פג
הגשת השגה למנהל הארנונה 90 יום מקבלת חשבון דדליין שלילי איבדת זכות להשגה לאותו חשבון
תגובת מנהל הארנונה 60 יום (90 עם הארכה) אישור־שתיקה ההשגה התקבלה אוטומטית
הגשת ערר לוועדת ערר 30 יום מקבלת דחייה דדליין שלילי איבדת זכות לערר
תגובת הרשות לוועדת ערר 30 יום מקבלת הערר שעון מנהלי אין סנקציה אוטומטית
החלטת ועדת הערר אין דדליין חוקי שעון מעקב תזכורות + פנייה לנציב
ערעור לבית משפט מינהלי 30 יום מהחלטת הוועדה דדליין שלילי איבדת זכות לערעור · נדרש עו"ד
למה זה חשוב מבין כל השעונים, רק אחד הוא "אישור־שתיקה" — תגובת מנהל הארנונה להשגה הראשונה (60 יום). זה ה-moat המשפטי הייחודי של המוצר. בכל שאר השלבים, שתיקה של הרשות היא לא כניעה אוטומטית — היא רק עיכוב מנהלי שמצריך מעקב ולחץ. ה-Claim Agent מבין את ההבחנה הזו ומפעיל את הכלים הנכונים בכל שלב.
המשך הסיפור הרוחב עם תוספת שכבת הערעור, התמונה השלמה היא: השגה → תגובה → במידת הצורך ערר → תגובה → במידת הצורך נציב/בית משפט. ב-90% מהמקרים זה ייגמר ברמת ההשגה (אישור או אישור־שתיקה). ב-8% נצטרך ערר. ב-2% בלבד — עו"ד אנושי. זה הופך את המוצר לאוטונומי בכ-98% מהמקרים.

השפעה על המודל העסקי

שכבת הערעור היא חלק מהשירות הבסיסי — לא תוספת. הלקוח שילם פעם אחת, ואם הרשות דחתה, אנחנו נלחמים בעבורו עד הסוף האוטונומי. רק במעבר לעו"ד אנושי (2% מהמקרים) — שם נדרש תשלום נוסף או הסכם נפרד.

זה הבדל מהותי מ"הגשה אוטומטית" של מתחרים: אצלם, דחייה = סוף הדרך. אצלנו, דחייה = תחילת הקרב.
