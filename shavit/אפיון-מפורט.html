1ארכיטקטורה ומבנה הפרויקט

המערכת נבנית בארכיטקטורה מודולרית תלת-שכבתית עם קוד משותף (Shared) בין הלקוח לשרת - כך שכל ישות, כלל ולידציה והרשאה מוגדרים פעם אחת ונאכפים בשני הצדדים.

Angular (SPA)ממשק עברית RTL רספונסיבי · רכיבי UI משותפים · Guards לניתוב
⇄
Remult APIישויות + ולידציות + הרשאות · CRUD מאובטח · Backend Prefilter
⇄
Node.js ServicesJWT · מתזמן (Cron) · דיוור · חתימת S3
⇄
תשתיותאירוח Railway (שרת + PostgreSQL) · AWS S3 · דיוור Resend

מבנה תיקיות (מודולריות מלאה)

src/
├── app/                          // צד לקוח - Angular
│   ├── common/                   // רכיבים משותפים: טבלה, צ'יפ סטטוס, דיאלוג, אשף, Drop-Zone

├── server/                       // צד שרת - Node.js

עיקרון המודולריותכל מודול (כיולים, לקוחות, תזכורות, קריאות…) מחזיק ישות, בקר ומסך משלו, ונשען על רכיבי ה-common. מודול עתידי - למשל קריאת שירות - נבנה על התשתית הקיימת (סוג + טופס ייעודי) ומבודד מהמודולים האחרים.

אכיפת הפרדת נתונים בשרת (Remult Backend Prefilter)

הסינון לפי תפקיד מוגדר על הישות עצמה ונאכף בכל בקשת API - לא ניתן לעקיפה מהדפדפן:

@Entity<Calibration>('calibrations', {
  backendPrefilter: () =>
    remult.user?.role === Role.CUSTOMER
      ? { customerId: remult.user.customerId }   // לקוח: אך ורק הרשומות שלו
      : {}                                        // מנהל/משרד: הכל
})
אותו עיקרון בכל הישויותMediaFile, Request ו-ReminderLog מסוננים באותה שיטה. מכייל שטח מקבל תצוגת רשימה מצומצמת ללא פרטי קשר מלאים של לקוחות.

5.1 · כניסה/loginציבורי
  • מסך ממותג: לוגו מאזני שביט, רקע כהה בשפת האתר, קישור לאתר החברה.
  • אימייל + סיסמה, "זכור אותי" פעיל כברירת מחדל - הלקוח מתחבר פעם אחת והחיבור נשמר (טוקן מתחדש).
  • שחזור סיסמה בקישור למייל; משתמש חדש נכנס לראשונה דרך קישור הזמנה וקובע סיסמה.
  • אחרי כניסה: חזרה ליעד המקורי אם היה (למשל קישור ישיר לתעודה), אחרת למסך ברירת המחדל לפי תפקיד.
5.2 · כיולים - המסך הראשי/calibrationsמנהל/משרד
  • שורת מדדים: סה"כ בתוקף · לקראת תפוגה (בתוך חלון המדרגה) · פגי תוקף · קריאות חדשות · לקוחות פעילים.
  • טבלה: לקוח, מכשיר, דגם, מס"ד, מס' תעודה, תאריך כיול, בתוקף עד, ימים נותרו, סטטוס (צ'יפ צבעוני), תעודה (צפייה/הורדה), פעולות.
  • חיפוש וסינון: טקסט חופשי, לקוח, סטטוס, טווח תאריכים; מיון בכל עמודה; עימוד לרשימות גדולות; ייצוא לאקסל.
  • כרטיס כיול (נפתח בלחיצה): כל פרטי התעודה, קבצים (PDF ללקוח + XLSX פנימי), היסטוריית גרסאות (תעודות שהוחלפו), תזכורות שנשלחו בגינו, סימון "תואם ידנית".
  • כיול ידני חדש: טופס הוספה של תעודה בודדת (כולל צירוף קבצים) - למקרים שוטפים בלי אשף ההעלאה.
5.3 · לקוחות/customersמנהל/משרד
  • טבלה: שם, איש קשר, אימייל (עם דגל אזהרה אם חסר), טלפון, מס' מכשירים, כיול אחרון, הקרוב לפוג, סטטוס.
  • כרטיס לקוח: פרטים + נמענים נוספים, רשימת המכשירים והכיולים שלו, תזכורות אחרונות, קריאות פתוחות, משתמש הפורטל שלו (+ שליחת הזמנה חוזרת).
  • הקמה: ידנית ("לקוח חדש") או אוטומטית בקליטה מרוכזת. מיזוג כפילויות שם בקליטה - עם אישור המשרד.
5.4 · העלאה מרוכזת (Bulk)/uploadמנהל/משרד

5.5 · השלמת פרטים לקליטהמנהל/משרד
  • טבלת עריכה מרוכזת לתעודות שנקלטו עם פרטים חלקיים: תאריך כיול, מס' תעודה, מכשיר, תוקף.
  • המערכת מציעה ערכים אוטומטית מתוך שם הקובץ לפי דפוס שמות אחיד (ייקבע מול דוגמאות אמת - ר' פרק 10); המשרד מאשר או מתקן.
  • עריכה ישירה בתאים, שמירה מרוכזת, סימון שורות שהושלמו.
5.6 · תזכורות ומעקב/remindersמנהל/משרד
  • רצועת מדיניות: המדרגות הפעילות (30 → 14 יום), שעת ריצה 11:00, קיצור למסך ההגדרות; כפתור "הרצת בדיקה" (תצוגה מקדימה של מי יקבל היום - בלי לשלוח).
  • טאב "קרובות": מי מתוכנן לקבל תזכורת בימים הקרובים - לקוח, מכשירים, תפוגה, מדרגה.
  • טאב "יומן שליחה": כל המיילים שנשלחו - סוג, נמען, מועד, סטטוס (נשלח/נכשל+סיבה), נפתח אישור או לא.
  • טאב "מעקב אישורים": חתך לפי מחזור - אישרו תיאום · לא הגיבו · תואמו ידנית · פג תוקפם ולא ביצעו. ייצוא כל חתך לאקסל.
  • פעולות שורה: סימון "תואם ידנית" (עוצר תזכורות למחזור), שליחה חוזרת, פתיחת כרטיס הלקוח.
5.7 · קריאות/requestsמנהל/משרד
  • רשימת כל הקריאות: סוג (כיול), לקוח, מקור (אזור אישי / מייל / משרד), מועד מועדף, הערות, סטטוס, מטפל.
  • עדכון סטטוס בזרימה: חדשה → בטיפול → תואמה → בוצעה; התראת מונה בתפריט על קריאות חדשות.
  • המסך גנרי לפי סוג - "קריאת שירות" עתידית תופיע כאן כסוג נוסף עם הטופס הייעודי שלה, באותו מסך.
5.8 · אזור לקוח/myלקוח
  • פתיח אישי: שלום [איש קשר], מדדים קטנים - המכשירים שלי, בתוקף, לקראת תפוגה.
  • הכיולים שלי: טבלת התעודות שלו בלבד - מכשיר, מס' תעודה, תאריך, בתוקף עד, סטטוס, צפייה/הורדת PDF (קישור חתום זמני).
  • קריאת כיול: טופס קצר (מועד מועדף, הערות) → הקריאה נרשמת ומופיעה למשרד; הלקוח רואה את הסטטוס שלה.
  • עדכונים: ציר אירועים - תעודות חדשות שהועלו, תזכורות שנשלחו, קריאות ועדכוני סטטוס.
  • אין ללקוח שום גישה לנתוני לקוחות אחרים, לקובצי עבודה פנימיים או למסכי ניהול - נאכף בשרת (פרק 4).
5.9 · אישור תיאום/confirm/:tokenציבורי בטוקן
  • נפתח מכפתור "אישור תיאום" במייל התזכורת - בלי צורך בסיסמה (טוקן חד-פעמי, תקף 30 יום).
  • מציג את המכשירים שבתזכורת + אפשרות לציין מועד מועדף והערה → כפתור "אני מאשר תיאום כיול".
  • האישור נרשם מיידית: נעצרות תזכורות המשך למחזור, נפתחת קריאת כיול למשרד, והלקוח מקבל אישור על המסך.
  • טוקן שכבר מומש מציג את סטטוס האישור הקיים (ללא כפילות).
5.10 · רשימת עבודה למכייל/techמכייל שטח
  • תצוגה בסיסית: כיולים קרובים ולקוחות לתיאום - תאריך, לקוח, כתובת, מכשירים.
  • מהווה בסיס להרחבה עתידית: הזנת תוצאות כיול מהשטח (טאבלט) והפקת תעודה במקום.
5.11 · משתמשים/usersמנהל
  • רשימה: שם, אימייל, תפקיד, לקוח מקושר (אם רלוונטי), פעיל/מושבת, כניסה אחרונה.
  • הוספת משתמש → נשלח קישור הזמנה במייל לקביעת סיסמה; השבתה מיידית חוסמת גישה.
  • יצירת משתמש-לקוח ישירות מכרטיס הלקוח בלחיצה אחת.
5.12 · הגדרות/settingsמנהל
  • תזכורות: ניהול המדרגות (הוספה/הסרה/שינוי - 30, 14, ואפשר גם 7 וכו'), שעת ריצה, מתג עדכון תעודה חדשה.
  • תבניות מייל: עריכת נושא וגוף לכל תבנית עם משתני שיבוץ + תצוגה מקדימה חיה ושליחת מייל ניסיון.
  • מיתוג: לוגו, צבעי מותג, קישור לאתר; משתקף מיידית בפורטל ובמיילים.
  • דיוור: שם וכתובת השולח; חיווי תקינות הדומיין (SPF/DKIM) לשיפור עבירוּת.

6מנוע התזכורות - אלגוריתם

משימת רקע יומית ב-11:00 (שעון ישראל, scheduler.ts). כל ריצה נרשמת ביומן עם תוצאותיה.

  1. איסוף: לכל לקוח פעיל - התעודה העדכנית של כל מכשיר (הגרסה האחרונה בלבד; תעודות שהוחלפו אינן נספרות).
  2. חישוב: ימים לתפוגה = expiryDate - היום.
  3. התאמת מדרגה: תעודה נכללת אם הימים לתפוגה נמצאים בחלון של מדרגה מוגדרת (30 / 14) וטרם נשלחה תזכורת מאותה מדרגה לאותו מחזור (בדיקה מול cycleKey ביומן). חלון "השלמה" תופס גם מקרים שחלפו בזמן השבתה - אף אחד לא נופל בין הכיסאות.
  4. סינון עצירות: מחזור שאושר תיאום (מהמייל או ידנית) או שנפתחה בגינו קריאת כיול - לא מקבל תזכורות נוספות.
  5. קיבוץ: כל התעודות הרלוונטיות של אותו לקוח מאוחדות למייל מרוכז אחד - לא מציפים את הלקוח.
  6. שליחה ותיעוד: שליחה עם טוקן אישור ייחודי; רישום מלא ב-ReminderLog כולל כשלים (כתובת שגויה, תיבה מלאה) שמוצגים למשרד לטיפול.
  7. לקוחות ללא אימייל: לא נשלחים - נאספים לרשימת חריגים שמוצגת למשרד בכל ריצה.

כללי יסוד

  • לא מטרידים פעמיים: אותה מדרגה לא נשלחת פעמיים לאותו מחזור - מובטח ברמת הנתונים, לא רק בלוגיקה.
  • אישור עוצר: אישר הלקוח תיאום - נעצרות תזכורות ההמשך של המחזור. תואם טלפונית? המשרד מסמן "תואם ידנית" בלחיצה - ואותו אפקט.
  • תעודה חדשה מאפסת: נקלטה תעודה חדשה למכשיר - נפתח מחזור חדש; התזכורות הבאות יחושבו מהתוקף החדש.
  • שקיפות מלאה: "הרצת בדיקה" מציגה מראש מי יקבל היום מייל; היומן מתעד כל שליחה בפועל.
  • מודעות למכסת דיוור: המנוע מכבד את מגבלת ספק הדיוור (100 ביום בתוכנית החינמית של Resend) - פריסת שליחות או התראה למשרד לפי הצורך (ר' פרק 7).

צפייה בתעודה

לידיעתך, כל תעודות הכיול שלך זמינות באזור האישי - כולל היסטוריית השנה האחרונה.

  • משתני שיבוץ: שם לקוח, איש קשר, טבלת מכשירים, תאריכים, קישורים - מוחלפים אוטומטית בשליחה.
  • מיילים נוספים במערכת: הזמנת משתמש חדש, שחזור סיסמה, אישור קליטת קריאה - באותה שפה עיצובית.
  • שליחה מכתובת רשמית של מאזני שביט; הגדרות SPF/DKIM לדומיין - כדי שהמיילים יגיעו לתיבה הראשית ולא לספאם.
  • ספק הדיוור - Resend: התוכנית החינמית מוגבלת ל-100 מיילים ביום (3,000 בחודש) - מספיקה לפעילות השוטפת הצפויה. בסבבי שליחה גדולים או עם הגידול: תוכנית Pro ($20 לחודש, 50,000 מיילים, ללא מגבלה יומית).
  • ויסות מכסה: תור השליחה מודע למכסה היומית - חריגה צפויה (למשל עדכוני "תעודה חדשה" אחרי סבב קליטה גדול) נפרסת אוטומטית על פני ימים או מוצגת למשרד להחלטה.

8קבצים וענן (AWS S3)

  • העלאה ישירה מהדפדפן: השרת מנפיק קישור חתום (Presigned PUT) לכל קובץ; הקובץ עולה ישירות ל-S3 בלי לעבור דרך השרת - מהיר, חסכוני ועמיד גם במאות קבצים במקביל.
  • הורדה מאובטחת: צפייה/הורדה בקישור חתום זמני (Presigned GET, תוקף קצר) - אין קבצים ציבוריים; הבקשה עוברת בדיקת הרשאה לפני הנפקת הקישור.
  • מבנה מפתחות:
shavit-calibrations/
└── customers/{customerId}/
    ├── certificates/{year}/{fileName}.pdf      // תעודות - גלוי ללקוח דרך המערכת
    ├── worksheets/{year}/{fileName}.xlsx       // קובצי עבודה - משרד בלבד
    └── reports/batch-{batchId}.csv             // דוחות קליטה
  • שרידות וגיבוי: S3 עם Versioning פעיל (הגנה מדריסה בטעות); בסיס הנתונים מגובה בגיבויים מתוזמנים מובנים של Railway - יומי (נשמר 6 ימים), שבועי (חודש) וחודשי (3 חודשים) במקביל. אין תשלום נפרד על הפיצ'ר - החיוב הוא על נפח האחסון האינקרמנטלי בלבד, זניח בסדרי הגודל של המערכת.
  • סדרי גודל: תעודת PDF ממוצעת ~0.5MB; אלף תעודות בשנה ≈ 0.5GB - עלויות אחסון זניחות (חשבון הענן ע"ח הלקוח, מפורט בהצעת המחיר).

9שכבת ה-API

רוב פעולות ה-CRUD מסופקות אוטומטית ע"י Remult על הישויות, עם ולידציות והרשאות מוגדרות בישות עצמה. בקרים ייעודיים (shared/controller) לפעולות מורכבות:

בקרפעולות עיקריותתפקיד

10קליטת נתונים היסטוריים (שנה אחורה)

  1. קלט מהמשרד: רשימת לקוחות (אקסל: שם, איש קשר, אימיילים, טלפון) + תיקיות התעודות של השנה האחרונה, מסודרות לפי שם לקוח - כפי שהן שמורות היום.
  2. טעינת לקוחות: ייבוא רשימת הלקוחות תחילה - היא הבסיס שמולו מוצלבות התיקיות.
  3. קליטת תעודות: דרך אשף ההעלאה במצב HISTORY - ללא שליחת מיילים ללקוחות.
  4. חילוץ מטא-דאטה: תאריך, מס' תעודה ומכשיר מחולצים משם הקובץ לפי דפוס השמות הקיים (ייקבע סופית מול מדגם קבצים אמיתי); מה שלא זוהה - מושלם מרוכז במסך "השלמת פרטים" (5.5).
  5. תוקף: ברירת מחדל 12 חודשים ממועד הכיול; חריגים נערכים פרטנית.
  6. אימות מדגמי: בדיקת נכונות שיוך על מדגם לקוחות יחד עם המשרד לפני הפעלת התזכורות.

מפתח ההחלפה (כפי שסוכם בפגישה)

כלל שנה+חודשמפתח כיול לקליטה: לקוח + שנה + חודש (periodKey). קובץ נכנס כשקיים כבר כיול באותו מפתח - מחליף אותו (הקודם נשמר כגרסה); אחרת - נוצר כיול חדש.
המלצת ספק לאישורללקוח עם כמה מכשירים המכוילים באותו חודש, מפתח של שנה+חודש בלבד יגרום להחלפות שגויות. ההמלצה: המפתח יכלול גם את מספר המכשיר - לקוח + שנה + חודש + מס"ד (או מס' תעודה). ההחלטה תיסגר מול דוגמאות אמת בתחילת הפרויקט (ר' פרק 13).

12פיילוט, הדרכה ומסירה

  • מסירה (סוף יום 10): מערכת מלאה בסביבת הענן, עם נתוני האמת שנקלטו (לקוחות + שנה אחורה של תעודות).
  • הדרכה (במסירה, לפני הפיילוט): מפגש מעשי לצוות המשרד (עד 5 משתתפים, כשעתיים) על המערכת החיה: קליטת תעודות, מעקב תזכורות, ניהול לקוחות וקריאות + מדריך תפעול מקוצר ומאויר.
  • פיילוט (ימי עבודה 11-15): עבודה שוטפת אמיתית בליווי צמוד - קליטת תעודות חדשות, ריצת תזכורות יומית, הפעלת אזור אישי ל-2-3 לקוחות נבחרים; תיקונים וכוונונים תוך כדי.
  • קריטריוני סיום פיילוט: תזכורות נשלחות אוטומטית ונרשמות ביומן · הצוות קולט תעודות עצמאית · לקוחות הפיילוט צופים ומורידים תעודות · אפס תקלות חוסמות פתוחות.
  • לאחר הפיילוט: עלייה מלאה לאוויר. ליווי ותחזוקה שוטפת - במסגרת הסכם נפרד, ככל שיסוכם.
